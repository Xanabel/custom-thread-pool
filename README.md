# Custom Thread Pool (XanaPool)

Дисциплина: Многопоточное и асинхронное программирование на Java

Тема курсовой работы: Реализация собственного Thread Pool

## Описание проекта

В рамках курсовой работы был разработан собственный пул потоков без использования стандартного класса ThreadPoolExecutor.

Основная цель проекта — изучение принципов многопоточного программирования, управления жизненным циклом потоков, распределения задач между рабочими потоками и обработки перегрузки системы.

Разработанный пул потоков получил название XanaPool.

## Балансировка нагрузки

Для распределения задач используется алгоритм Round Robin.

Каждая очередь закреплена за отдельным рабочим потоком.

Новая задача направляется в следующую очередь по кругу:

Queue0 → Queue1 → Queue2 → Queue3 → Queue0 ...

Преимущества данного подхода:

простота реализации;
равномерное распределение нагрузки;
отсутствие концентрации задач в одной очереди.

### Work Stealing

Если рабочий поток не обнаружил задач в своей очереди, он пытается
"украсть" задачу из очереди другого потока (метод stealTask).

Поток последовательно проверяет все чужие очереди неблокирующим
способом (poll без ожидания) и забирает первую найденную задачу.

Это позволяет:
- избежать простоя потоков при неравномерной загрузке очередей;
- повысить утилизацию потоков;
- уменьшить количество отклонённых задач.

## Политики отказа

В проекте реализованы две политики обработки перегрузки.

### AbortRejectedTaskHandler

Если очередь заполнена и достигнуто максимальное количество потоков, задача отклоняется.

Пример:

[Rejected] Task Request-21 was rejected due to overload!

Преимущества:

защита системы от перегрузки;
отсутствие бесконтрольного роста памяти.

Недостатки:

часть задач может быть потеряна.

### CallerRunsRejectedTaskHandler

При перегрузке задача выполняется в вызывающем потоке.

Преимущества:

задача гарантированно выполняется.

Недостатки:

поток клиента может быть заблокирован длительным выполнением задачи.

## Логирование

В проекте реализовано логирование следующих событий:

- создание потоков;
- поступление задач;
- выполнение задач;
- отклонение задач;
- завершение потоков;
- завершение работы пула.

Пример:

```
[ThreadFactory] Creating new thread: XanaPool-worker-1
[Pool] Task accepted into queue #1: Request-5
[Worker] XanaPool-worker-2 executes Request-5
[Rejected] Task Request-21 was rejected due to overload!
```

## Завершение работы пула
### shutdown()

После вызова метода:

- новые задачи не принимаются;
- уже поставленные задачи выполняются до конца;
- после обработки очередей потоки завершаются корректно.

### shutdownNow()

После вызова метода:

- очищаются очереди задач;
- рабочие потоки прерываются;
- выполнение оставшихся задач прекращается.

### awaitTermination()

Метод блокирует вызывающий поток до тех пор, пока:
- все потоки пула не завершатся после shutdown;
- не истечёт указанный таймаут;
- вызывающий поток не будет прерван.

Возвращает true, если пул полностью завершился, и false — если истёк таймаут.

## Мини-исследование параметров

Во время тестирования были изменены значения размеров очередей и количества потоков.

### Маленькая очередь

Плюсы:

быстро обнаруживается перегрузка;
меньше потребление памяти.

Минусы:

больше отклонённых задач.

### Большая очередь

Плюсы:

меньше отказов.

Минусы:

увеличивается время ожидания задач.

### Большое количество потоков

Плюсы:

повышается параллелизм.

Минусы:

растут накладные расходы на переключение контекста.

Для демонстрационного сценария оптимальными оказались параметры:

corePoolSize = 2
maxPoolSize = 4
queueSize = 5
